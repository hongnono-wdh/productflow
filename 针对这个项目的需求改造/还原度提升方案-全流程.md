## §0 v2 相对 v1 的关键修订（诚实变更记录）

**两处推翻（v1 判断被行业证据否定）：**

1. **❌ 像素相似度% 硬闸门 → 改为「三层闸门」。** v1 想「pixelmatch/SSIM 算相似度%、低于阈值卡 done」。调研 4 证明这是**拿噪声比噪声**：设计稿本身 AI 生成不精确（非金标准）+ 实现有 2–3% 天然渲染噪声（字体/抗锯齿）+ 两者异源不可像素对齐（一个 4px 位移让每条边变红）。且 LLM 当「绝对打分器」（输出「87.3%」）也不可靠（有系统偏差、会漏「不可命名的差异」）。**改为：LLM 视觉裁判出分档 pass/fail + 差异清单（硬 gate）／像素差异热力图作证据（软信号）／DOM·文本断言（正交硬 gate）。**
2. **❌ 营销页整页生图 → 改为「骨架组件化 + 只生图素材」。** 调研 3 指出整页生图会**遗传** AI 临摹还原度病、改文案即需重生图、token 漂移。行业主流是「系统化到 token/组件层，独特性放在视觉资产层」。**改为：营销页排版/CTA/栅格用组件+token，只把 hero 大图/背景/氛围图交给 gpt-image 生成再嵌入。**

**主要升级：**

- **PNG 降级为预览、结构化 spec 升为主线**（调研 5：业界严肃工具无一靠「喂图临摹」，全是「结构在先、像素在后」）。**渐进落地**：④ 仍出 PNG 预览 + 额外产结构化 sidecar，⑥ 主读 sidecar（决策见 §0 末）。
- **design-spec 对齐 DTCG 骨架 + 两层 token 为主 + 自研 Python 编译到三端**（调研 2）。
- **⑥ 采用分治生成 + 带护栏的视觉自纠闭环**（调研 5：整图临摹错误 85% 是元素遗漏；自纠有效但 3–4 轮衰减、需防 reward hacking）。
- **9 态矩阵做成设计阶段强制 gate**（调研 1/3：漏状态是还原度最大杀手，而 AI 被强制反而不会漏）。
- **映射表 + 确认门 + 组件目录**（调研 3：Halodoc 生产案例，架构与 ProductFlow 同构，80–90% 保真）。

**已拍板的 4 个关键决策（2026-07-03）：**

| 分叉                | 决策                                                                                             |
| ------------------- | ------------------------------------------------------------------------------------------------ |
| PNG 降级程度        | **渐进**：PNG 预览 + 结构化 sidecar，⑥ 主读 sidecar；sidecar 结构留「将来升为主产物」的路 |
| 营销页              | **骨架组件化 + 只生图素材**（接受行业纠正）                                                |
| token 编译工具      | **自研轻量 Python 编译器**（纯标准库、零依赖，契合项目铁律）                               |
| ⑥ 还原度验收深度   | **最深档**：三层闸门 + **3 轮自动自纠** + 人工圈选兜底                               |
| ④ 设计范式（前定） | **混合**：产品 UI 组件库代码化，营销页按上面新规则                                         |

## §1 根因诊断（证实并深化）

**还原度差非 ⑥ 不努力，而是全流程从未创建「精确、机器可照抄的视觉规格」；视觉信息一路以模糊文字/像素传递，每过一道 AI 生图蒸发一次精确性。**

本质是**「结构 → 像素 → 结构」的有损往返**：③④ 把设计意图压成 PNG 像素，⑥ 再让另一个 AI 从像素有损解压回结构。学术归因（DCGen）量化了代价：整图直出的错误里 **85.34% 是「元素遗漏」**（整页超出 MLLM 推理容量）。Figma 官方立场一针见血：**「正确的代码对齐的是设计意图，不是像素。」**

四个泄漏点（一手证据见各阶段手册 + `pf_state.py`）：L1 ②参考图未萃取 token；L2 ③styleSummary 无结构、精确 prompt 只当对话记录；L3 ④设计稿是不可落地的 gpt-image 幻想 PNG；L4 ④→⑤断层 + UI 状态（空/载/错/hover）从没人定义。

## §2 五条心法（衡量任何改动的尺子）

1. **精确前置（shift-left）**：精确视觉规格尽量往上游推，从 ② 就从真实参考萃取，别拖到 ④ 才半成型。
2. **结构化沉淀（单一事实源）**：视觉信息全程以机器可读 token 存在、逐级加精，不走自由文本。`#3498db` 不会被不同 agent 不同理解。
3. **同源实现（design-dev parity）**：设计与开发共用组件库 + token。设计稿不是「要临摹的画」，是「用同一套积木搭的、可复用的东西」。
4. **可验证闸门**：每个交接点有可对齐检查，⑥ 用三层闸门 + 自纠闭环收敛。
5. **结构优先于像素（structure-over-pixels）**：全行业最强共识。宁可传结构化描述让 AI 照抄，不传像素图让 AI 临摹。PNG 是预览，不是事实源。

## §3 统领设计：还原度脊椎 `design-spec`（v2 升级版）

一根贯穿全程的**结构化视觉规格**，每阶段往里填充/加精，逐级从模糊到精确：`①萌芽→②萃取→③加精→④定稿→⑤补全→⑥引用`。它是 profile 信号架构在**视觉维度**的具体化，复用同一套 `state.json` 挂载 / `pf_state` CLI 读写机制。

**v2 三条关键工程约定（源自调研 2）：**

**(a) 对齐 DTCG 骨架，但用简化 value。** token 用 `{ "$value": "#3498db", "$type": "color" }`、`{ "$value": "16px", "$type": "dimension" }`；别名用 `"$value": "{color.blue.500}"`。**不采用** DTCG 2025.10 的 color/dimension 对象格式（`{colorSpace,components}`）——工具链自己都还没 full support，扁平字符串 AI 照抄直接可用、少一层出错面。

**(b) 两层 token 为主（primitive + semantic），component 层按需。** primitive 纯描述（`color.blue.500`、`space.4`、`radius.md`）、semantic 表意（`color.action.primary`、`text.body`）；component（`button.bg`）只在需要单独调时才加，别一次性铺满（避免「过度抽象」坑）。dark/主题只在 semantic 层 override。

**(c) 自研轻量 Python 编译器，一份 token → 三端。** 纯标准库写一个小编译器，把 `design-spec.tokens` 编成 Web `--css-var` / iOS `Color(hex:)` / Android Compose `Color(0xFF..)`。**让「三端一致」成为编译产物而非 AI 自觉**——AI 没 code review 兜漂移，比人更需要这条机器保证。（放弃 Style Dictionary，因其 Node 依赖违背项目「纯 Python 零依赖」铁律。）

**结构草案（字段随阶段从空到满）：**

```jsonc
{
  "componentLib": { "PC":"shadcn/ui+tailwind", "H5":"shadcn/ui+tailwind",
                    "APP":"SwiftUI(HIG) / Material3(Compose)" },   // ② 锁定；⑥ import
  "tokens": {                                    // DTCG 骨架 + 简化 value；两层为主
    "color": { "blue":{"500":{"$value":"#3498db","$type":"color"}},
               "action":{"primary":{"$value":"{color.blue.500}","$type":"color"}} },
    "space": { "4":{"$value":"16px","$type":"dimension"} },
    "radius":{ "md":{"$value":"8px","$type":"dimension"} }
  },
  "pages": [                                      // ④ 填组件构成 + 映射；⑤ 填状态矩阵
    { "id":"pg-home", "type":"marketing|product",
      "components":[{"slot":"hero.cta","lib":"Button","variant":"primary"}],
      "states":{ "CTAForm":["default","submitting","error","success"] },
      "assets":[{"slot":"hero.bg","gen":"gpt-image","file":"..."}] }   // 营销页只生图素材
  ],
  "provenance": { "refs":["ref-a1"], "hero":"...", "heroPrompt":"..." }  // 可追溯
}
```

**PNG 的新定位（决策：渐进 sidecar）**：④ 仍出 PNG 给人预览/审批，但**额外产 `design-spec` 的 `pages[]` 结构化 sidecar**；⑥ **主读 sidecar、PNG 作辅助参照**。sidecar 结构即上面的 `pages[]`，设计成「将来能平滑升级为主产物」——先渐进，架构留通向「彻底 spec-first」的路。

## §4 全流程还原度地图 R-①~⑥（并入决策与行业证据）

### R-① 市场调研

- `R-①a` 截存竞品关键页真图当**原始像素锚点**（`artifacts/phase-1/refs/`）。
- `R-①b` 按产品类型**萌芽候选组件库家族**（SaaS→shadcn/Material；营销→偏定制少组件）。
- `R-①c` 起 `design-spec.json` 骨架（空 tokens + 候选库）。

### R-② 找参考 ⭐ 高杠杆、建议先做

- `R-②a` **用视觉能力从选中参考图萃取结构化 token**（hex/字体气质/间距节奏/圆角阴影）→ `refs[].spec` 汇总进 `design-spec.tokens` 首版。把「最有参考价值的真实像素信息」从图里救出来（补 L1）。
- `R-②b` **锁定各平台组件库 + 基础主题**，产出**机器可读组件目录草案**（Halodoc `DS_AGENTS.md` 式：组件名 + API + 适用场景）。参考风格→匹配最近成熟库。

### R-③ 首图设计

- `R-③a` styleSummary **结构化**为 `design-spec.tokens` 字段（补 L2）。
- `R-③b` 首图定稿后**反向萃取精确 token**（vision 读定稿图取真实 hex/字体/圆角/阴影），把「AI 画的基调」固化成「可照抄的数值」。
- `R-③c` 基调色板 → 覆盖 ②b 组件库 theme；prompt 存 `provenance` 留痕。

### R-④ 页面设计 ⭐ 核心（重构）

- `R-④a` **direction.md 升级为 `design-spec.tokens` + 组件库 + 主题 override**（机器可读，替代「圆角约 4px」这类文字）。
- `R-④b` **产品 UI 走组件库映射**：出「设计元素→组件库组件」**映射表**（`pages[].components`）；**PNG 仍出作预览，额外产结构化 sidecar**（决策：渐进）。⑥ 照 sidecar 拼装、不反推图。
- `R-④c` **营销页：骨架组件化 + 只生图素材**（决策）——排版/CTA/栅格用组件+token，只把 hero 大图/背景/氛围图交 gpt-image 生成（记 `pages[].assets`）再嵌入。
- `R-④d` **9 态交互矩阵做成强制 gate**（缺态阻断）：default/hover/focus/active/disabled/loading/error/empty/skeleton。AI 被强制生成全态反而比人不漏（调研 1/3）。

### R-⑤ 功能与数据设计

- `R-⑤a` **数据态矩阵**：为关键页/组件补空数据/多数据/长文本截断/分页边界 → `pages[].states`（交互态在 ④，数据态在 ⑤）。
- `R-⑤b` 数据边界 → UI 边界（列表条数/字段长度/可空），避免 ⑥ 用假数据撑出跑版布局。
- `R-⑤c` ⑤ 反向校验 ④：确认组件够覆盖所有功能所需状态，缺则回 ④ 补。

### R-⑥ 开发实现（重构，验收最深档）

- `R-⑥a` **import ④ 组件库 + 套 `design-spec.tokens`**（自研编译器已把 token 编到三端），不肉眼猜（L1 兑现）。
- `R-⑥b` **分治生成**：按组件/区块粒度生成 + 对比，**不整页临摹**（DCGen：整图 85% 遗漏，聚焦即好转——最小投入最大收益）。
- `R-⑥c` **统一截图基准 + 环境确定性**（调研 4 最高杠杆）：固定 headless chromium 版本 / iOS 机型 / Android AVD+DPI / 视口；截图前 `document.fonts.ready`、禁动画、mask 动态内容。
- `R-⑥d` **三层闸门**（替换 v1 像素%）：① **LLM 视觉裁判**喂「设计稿+实现+checklist」出**分档 pass/fail + 差异清单**（硬 gate，不出百分比）；② **归一化对齐后的 pixelmatch 差异热力图**作证据（软信号，兜 LLM 结构盲区）；③ **DOM/文本断言**关键文案·必需区块·ER 字段落地（正交硬 gate）。
- `R-⑥e` **带护栏的自动自纠闭环（3 轮）**（决策 + 调研 5 护栏）：生成→截图→裁判→自动改→再来，**护栏**：(a) 渲染跑不起来=硬失败不进对比；(b) 只接受「确有改进」的修订；(c) **3 轮硬上限**；(d) 对比用可量化信号 + 裁判，不靠肉眼「差不多」。
- `R-⑥f` **人工圈选兜底**——已发版（P6-5 + 2.26.1 设计图随圈选过河）✓，作为自纠闭环之后的最后一道人工关。

## §6 尚待实施期敲定的细节（决策已定的不再列）

- **组件库具体选型**：Web shadcn/ui+Tailwind（默认）；iOS 只用 SwiftUI 系统组件+HIG（不引三方，默认）；Android Material 3 Compose（默认，Theme Builder 可直出 `Color.kt`）；桌面复用 Web 库。——默认值如上，第一期确认。
- **LLM 视觉裁判的 checklist 轴**：布局结构/主色配色/字体层级/关键组件存在/文案一致——具体轴与分档标准第四期定。
- **token 萃取**：纯 Claude 读图萃取 vs 加辅助；是否要人工确认关口。
- **9 态 gate 分级**：静态展示区块是否豁免部分态（按组件类型分级）。
- **降级路径**：无视觉能力/无组件库环境的 agent 如何优雅降级（沿用「方法论名词降级」惯例）。
