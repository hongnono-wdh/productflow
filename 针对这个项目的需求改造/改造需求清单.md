## 二、逐项清单（按流水线阶段）

> 每项格式：**现状 → 诉求 → 初步方向（待展开）→ 讨论钩子**。编号 `P{阶段}-{序号}`，并标来源图号与主线。

### ① 市场调研

#### 📝 P1-1 · 需求确认应追问关键可行性问题〔方案已定·代码待实施〕　〔image1｜主线 B｜P1〕- 特定领域下AI能够帮助用户提出隐藏性的问题 - 低优先级

- **现状**：brief 确认环节只问「领域偏好 / 转化动作 / 内容定位 / 推送频率」这类**偏好型**问题。
- **诉求**：应像产品经理一样**追问关键可行性/合规问题**——「AI 新闻数据从哪来？抓第三方数据是否合规？如何在合规前提下获取？」在确认阶段就把「能不能做、怎么合规做」挖出来。
- **初步方向**：在 brief 生成的追问清单里，针对产品类型补一类「数据来源 / 合规 / 关键依赖」的可行性问题。
- 💬 **讨论 / 方案（v1 已敲定 2026-06-29）**：把 brief 确认从「agent 自由生成 ≤6 个浅问题」升级为「**分型推断 + 可行性追问**」三段式。

  **① 分型：agent 推断 + 展示 + 可对话调整**

  - brief agent 读描述后，先推断 **产品类型**（内容/资讯 · 工具 · 社交UGC · 电商交易 · SaaS/B端 · 落地页…）+ **用户目标**（快速验证 waitlist · 做真能用的产品 · 复刻竞品 …）。
  - 推断结果**显式展示**给用户（如「判断：内容/资讯类 · 目标=做真能用的产品」），**纯 agent 推断、不让用户开场填表**。
  - 用户可**继续对话调整**——复用已有的多轮澄清管道（`clarify`），判错了说一句即可让 agent 重判。

  **② 追问：一次性给全部、用户自选、支持单/多选**

  - agent 按推断的产品类型，从**可行性框架**里挑出对这类产品最关键的问题，**一次性全部反馈**（不分轮渐进）。
  - 每个问题标注**单选 / 多选**（agent 判定），用户自己挑哪些答、怎么答。
  - 选项尽量做成「**带后果的智能选项**」（如数据来源选「公开抓取」时提示合规风险）。

  **③ 深度边界：聚焦「能不能做出来」（可行性）**

  - 追问维度收敛为可行性 + 必要的形态/范围：**数据/内容来源**（自有 / 第三方 API / 公开抓取 / UGC）、**合规/版权/隐私/牌照**、**技术依赖**（外部 API / 模型 / 能力，有无卡脖子）、**关键成本**（API/算力/数据采购）、**形态/范围**（产品形态、MVP 做什么/不做什么）。
  - **不做**：商业模式 / 变现 / 竞争 / 壁垒 / 融资 这类「值不值得做」的创业向问题。

  **落地改动（待实施）**：

  - `server.py:_auto_gen_brief` prompt 重写：去掉「落地页中心」硬假设（`scope=落地页范围`→中性产品范围）；加分型推断（输出 `profile` 字段）+ 可行性追问框架；`questions` 每条加 `multi` 单/多选标记；放开 ≤6 条上限。
  - `brief.json`：`questions` 增 `multi` 字段；新增 `profile`（分型推断：`type` + `goal`）。
  - 前端 `BriefPanel`：展示分型推断（可对话调整）；`questions` 支持多选作答 + 多选答案累积进澄清对话。
  - 守边界：**全自动模式**仍不打断（分型 + 假设显式写进 `positioning.md` 并标「未确认」）；整体保持「一句话起步」的轻量。

  **跨项协同**：`profile` 的「产品类型 + 平台」推断应**沉淀为贯穿全流水线的信号**——下游 P2-2（找参考关键词）、P4/P6（多平台出稿/预览）、P5-3（按平台出数据层）都复用它，与**主线 A** 协同。

  | 每页版本       | 页面卡片内部（现状不动）              | 卡片上切候选稿                                                 | 节点用该页当前版本缩略图                                   |

  **流程图构成（采纳确认 1–6）**：节点=页面（缩略图复用设计稿 + 页面名，可点高亮/跳该页，无图显**占位框**=该平台内某页还没出图）；边=`{从页, 到页, 操作标注}`，允许一页多出边 / 两页间多边，标一个**入口页**；**每平台各维护一套自己的边**（不跨平台共享）；生成=agent 为**每个平台**据「该平台页面清单 + 每页内容 + 功能模块」自动画线，用户可加/删边 · 改标注 · 拖节点 ·「重新生成」。

  **方案 B 的取舍**：每平台独立规划自己的页面清单与流程边 → 各平台**页面数量天然不同**（如「设置」：APP 拆 4 页 / PC 1 页带侧边 tab）、跨平台对应关系松。**好处**：数据解耦、实现简单（无需跨平台逻辑身份映射、无需占位兜底）。**代价**：跨平台一致性弱 → 靠**共享的「功能模块清单（④/⑤）+ 设计系统（P4-6）」兜底**：各平台虽独立拆页，但都从同一份功能模块 + 同一设计系统出发，不至发散。

  **数据结构（`canvas.json` stage-4 段扩展）**：按平台分组 `perPlatform{ PC|H5|APP: { pages（含每页多版本图）, gridLayout（平铺坐标）, flow{ edges, entry, nodePos } } }` + 顶层 `viewState{ mode, platform }`。每平台一份独立的「页面 + 两套布局 + 流程边」。

  **其它**：「整理排布」按钮在两模式下语义不同（平铺整理 / 按流程方向布局）；阶段名**保留「页面设计」不强改**（轻方案）；与 P4-7（已搁置）解耦、不喂 ⑥（仅给人看）。

  **关联**：左侧「平台分组 + 按平台比例出稿」同时落地 **P4-1 / 主线 A**。
