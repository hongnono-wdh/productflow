## 一句话目标

把 brief 确认从「agent 自由生成 ≤6 个偏好型浅问题、且 prompt 默认落地页」升级为「**先分型、再按产品类型追问可行性**」。

---

## 1. 分型 —— agent 推断 + 展示 + 可对话调整

- agent 读产品描述，推断 **产品类型**（内容/资讯 · 工具 · 社交UGC · 电商交易 · SaaS/B端 · 落地页…）+ **用户目标**（快速验证 waitlist · 做真能用的产品 · 复刻竞品 …）。
- 推断结果**显式展示**给用户（如「内容/资讯类 · 做真能用的产品」）；**纯 agent 推断，不让用户开场填表**。
- 用户可**对话调整**——复用现有 `clarify` 多轮澄清管道，判错说一句即可让 agent 重判。

---

## 2. 追问 —— 一次性全给 + 单/多选

- 按推断的产品类型，从**可行性框架**挑出**最关键**的问题，**一次性全部反馈**（不分轮渐进）。
- 每题标 **单选 / 多选**（agent 判定），用户自己挑哪些答、怎么答。
- 选项做成「**带后果的智能选项**」（如数据来源选「公开抓取」→ 提示合规风险）。

---

## 3. 边界 —— 聚焦「能不能做出来」（可行性）

- 追问维度收敛为可行性 + 必要的形态/范围：
  - **数据 / 内容来源**：自有 / 第三方 API / 公开抓取 / UGC
  - **合规 / 版权 / 隐私 / 牌照**
  - **技术依赖**：依赖哪些外部 API / 模型 / 能力，有无卡脖子
  - **关键成本**：API / 算力 / 数据采购
  - **形态 / 范围**：产品形态、MVP 这次做什么 / 不做什么
- **不做**：商业模式 / 变现 / 竞争 / 壁垒 / 融资 这类「值不值得做」的创业向问题。
