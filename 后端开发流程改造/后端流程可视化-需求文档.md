## 0. LLM 使用协议（READ FIRST）

本文件是一份**需求规格活文档**，供 AGENT 读取并据以工作，供 HUMAN 逐步补充。约定：

2. **ID 引用**：一切引用用稳定 ID——原则 `PRIN-n`、需求 `REQ-n`、决策 `DEC-n`。
3. **状态语义**：
   - 需求 `status: needs-confirmation` = agent 的解读**尚未经 HUMAN 确认**，不可当事实。`confirmed` 才是共识。
   - 决策 `status: open` = 未拍板；AGENT **不得**在 open 决策上自行选定后实施。
   - 字段 `agent_interpretation` 一律视为**提案**；字段 `user_verbatim` 是 HUMAN 原话、**唯一 ground truth**。

## 1. 项目背景（grounding，自包含）

- **ProductFlow** 是一套 Claude Code skill + 本地操作台（`localhost:7717`），把「做一个互联网产品」变成 **7 阶段流水线**：`①市场调研 → ②找参考 → ③首图设计 → ④页面设计 → ⑤功能与数据设计 → ⑥开发实现 → ⑦部署上线`。仓库本身是工具，不是用它产出的产品。
- **架构模型**：双入口（CLI 主通道 + 网页操作台）+ 单一状态源 `<project>/.productflow/state.json`。AGENT 执行流水线，每步用 `pf_state.py` 写状态，操作台经 WebSocket 实时呈现。
- **本需求作用域 = ⑤⑥ 阶段**，权威定义见 `productflow/scripts/pf_state.py` 的 `PHASES`：
  - **⑤ 功能与数据设计** steps：`module-list`(功能模块清单) → `er-diagram`(ER图) → `schema-ddl`(表结构DDL) → `api-contract`(API契约) → `pick-template`(选定开发模板)。手册：`productflow/references/phase-5-spec.md`。
  - **⑥ 开发实现** steps：`scaffold`(脚手架) → `frontend`(前端) → `backend`(后端) → `testing`(测试) → `api-docs`(接口文档)。手册：`productflow/references/phase-6-implement.md`。
- **关键代码锚点**：
  - 状态机 CLI：`productflow/scripts/pf_state.py`（`cmd_artifact` 登记产物、`cmd_page` 页面地图、`cmd_step`/`cmd_phase`）。
  - 操作台后端：`productflow/scripts/server.py`（`_auto_stage` 跑阶段、`_imagegen_key_ready` 生图 key 硬闸、`/api/deploy-creds` 部署凭证）。
  - 前端产物渲染：`web/src/components/Modal.tsx`（已支持 markmap 思维导图 + mermaid 图渲染，vendored、端上零 Node）、`web/src/components/Canvas.tsx`（③④ 画布）。
  - 现有可视化范式：④ 业务模块架构树 = markmap 渲染 `artifacts/phase-4/module-arch.mm.md`；⑤ ER 图 = mermaid 渲染 `artifacts/phase-5/er.md`。**本需求的新图应复用「agent 生成 .md 产物 → 操作台渲染」这条通路。**

## 3. 核心原则（Principles）

### PRIN-1 · 业务优先（先业务流程，再接口）
- **status:** confirmed（会议已达成的结论）
- **statement:** 后端开发**先定业务流程 / 领域模型，再导出接口**。接口是「前端 ↔ 后端」的契约，是业务流程在系统边界上的投影，跟着流程走。
- **canonical_order:** `业务流程 + 前端页面需求 → 汇出接口契约 → 展开后端执行流程 → 写代码`
- **rationale:**
  - 先有流程/数据/状态，接口才有稳定依据；「先拍接口再补流程」会让后端逻辑被接口形状绑架、难重构。
  - 接口非纯后端产物，同时被前端页面调用需求塑形；实践上迭代校准，但**主导权在业务流程**。
- **现状符合度:** ⑤ 顺序（模块清单→ER→表结构→API契约→选模板）本就是「流程/数据在前、接口在后」，方向正确。本需求补的是**接口之后、写代码之前**缺的一层：后端执行流程/架构可视化。

### PRIN-2 · 交互式关联 + 流程与数据流转打通（核心形态 · HUMAN 2026-07-02 澄清）
- **status:** confirmed
- **statement:** 本需求核心是**开发流程可视化**，且成品必须是**可交互关联**的、而非孤立静态图。在操作台页面上：
  - **可关联 / 联动**：节点（页面 / 模块 / 接口 / 数据表）可点击、可交叉跳转，彼此显式关联；
  - **流程与数据流转打通**：执行 / 开发流程要连到数据流转——从一个流程节点可追到它**读 / 写哪些数据**、连到数据模型；数据在各步骤间如何流转是**连贯、可追溯的一张图**。
- **形态类比:** 更接近一张**可交互的关联图 / 数据流图**（节点 = 页面 / 模块 / 接口 / 表，边 = 调用 / reads_from / writes_to / transforms），类似 understand-anything dashboard 那种可点、可下钻的图，但对象是**被生产的产品**。
- **implication:** 抬高了 DEC-1「载体」的门槛——纯静态 .md 图片（markmap / mermaid）不足以支撑「页面关联 + 数据可通」，需前端加**交互关联层**或走可交互视图。

### PRIN-3 · 人设计流程、agent 执行成品（运作模型 · HUMAN 2026-07-02 澄清）
- **status:** confirmed（HUMAN：「我就是后端同事，这个后端开发流程我来设计，然后有 agent 按设计的流程开发推进、最终形成完整产品」）
- **statement:** **双角色分离**——**HUMAN（后端同事）设计**后端开发流程（模块拆解 / 接口 / 数据流 / 开发顺序与门控）；**AGENT 按该设计执行**：跟着流程一个模块一个模块地开发推进（实现 + 测试 + 门控），最终形成完整产品。
- **implication:** 本设计文档是**双用途**——既是给人看 / 审的可视化设计，又是 **agent 可直接执行的流程规格**（呼应「文档写成 LLM 能理解」）。REQ-1/2/4（可视化）与 REQ-7（模块并行开发编排）是同一张「**可执行流程设计**」的两面：**人设计、agent 跟随执行**。

## 5. 原始需求（user_verbatim · ground truth · 请勿改写）

> 逐字保留 HUMAN 在会议/对话中的原话，供 AGENT 对照，避免解读漂移。

**核心问题：** 「后端开发，是先定流程业务再定接口，还是先定接口再定流程。」 → 结论见 PRIN-1。

**总述：** 「需要完善前端和后端的流程逻辑，已有前端流程，需要给到用户（我，技术人员）可视化的后端执行流程。」

**六点：**
1. 产品流程图，业务逻辑图，数据关系，数据模型图。
2. 可视化后端实现流程（大黑盒，拆成小黑盒）。
3. 如果有涉及到第三方的key，那么在页面上体现出来让用户选择。
4. 从产品设计之初和前端页面，完整流程，后端流程可视化设计。
5. 完善测试用例以及相关文档。
6. 项目框架在做之前agent直接调研，要求通用型！！！最高的（比如根据项目场景不同，去做不同的项目调研，默认参数---）。

**§5.7 追加（2026-07-02）：** 「然后实际开发的时候，开发过程控制为一个模块一个模块的开发、顺序执行，完成一个模块之后才开启下一个模块。」

**§5.8 追加（2026-07-02）：** 「项目开发完成后，在项目目录需要放项目相关文档，比如开发说明、使用说明、设计方案以及接口文档等。」

**§5.9 追加（2026-07-02）：** 「在后端开发之前需要问一下用户采用什么数据库，还是由 agent 自己定。」

## 7.6 细则决议（open_questions 收口 · 2026-07-02 · HUMAN 批准）

> §7.5 的 🟡/🟠 项与各 REQ `open_questions` 逐条定案。

**可视化**
- **A1**（REQ-1 表示法）：产品流程图 = mermaid flowchart；业务逻辑图 = 状态图；数据关系/模型 = 复用 ⑤ ER。可下钻。
- **A2**（REQ-2 粒度）：单接口 = flowchart(校验→业务→数据→外部)；时序图/调用链**按需**；错误分支只画关键失败路径、不穷举。

**关联 / 判定**
- **B1** 页面↔模块 = **N:N**，双向高亮，关联存 `backend-flow.json`；映射由 **agent 按页面内容 + 模块推断并登记**，**可在流程页框选纠正**（#5）。
- **B2** 涉后端判定 = **项目级开关 + 模块级标注**（纯前端模块标记、无后端节点）。
- **B3** 模块依赖：`backend-flow.json` 记依赖边，仅用于**集成/联调排序 + 影响面**；开发仍全并行。

**变更 / 一致性**
- **C1** 变更传播：契约 + schema = 单一事实来源；改动 → 顺依赖找受影响(模块 + 前端 mock + ORM/DDL/ER) → 自动更新 + 重跑 + 评审；图上标记受影响节点。
- **C2** schema：一份**全局 schema**，各模块并行声明 → 集成合并(冲突由整体评审 + 交互裁决)；migration 记录，简单项目可重建。
- **C3** 跨模块回归：集成时 + 每次交互改后跑**全量回归**，纳入整体流程评审。

**其它 REQ**
- **D**（REQ-3 key）：预置常见类型模板(API key/OAuth/DB URL/短信/支付/地图…) + 提示免费/付费/额度；「开发的 key」= 产品运行依赖，**不含生图 key**(归生产用/P4-5)。
- **E**（REQ-5 测试文档）：测试 = 单元 + 接口 + e2e(沿用 ⑥ 四类)；文档 = API 文档 + 架构说明(系统流程图导出) + 交付文档。
- **F**（REQ-6 选型）：默认参数 = 调研推荐配置(版本/目录/关键依赖)；**加按场景缓存**(复用 + 增量)解决慢/不稳定/难复现。
- **G**（REQ-7 评审判据）：判据 = 代码审查✔ + 测试✔ + 契约校验✔；覆盖率给建议阈值(关键路径)、不硬卡。**集成无人工终审闸**——对齐既有 ProductFlow（既有无强制终审，靠自动验证 + `choice` 关键点 + 全自动可跳的阶段确认）；本需求集成 = agent 自动整体流程评审，有要改随时框选直接改、不单独审。

**零碎**
- **H1** 横切关注点(错误/日志/配置) → 归入「地基」(scaffold + 公共层，模块前先建)。
- **H2** 反哺 ④：后端发现 ④ 漏页面 → agent 在**页面地图补占位 + 提示用户**，不擅改 ④ 设计稿。
- **H3** 断点续跑：`pf_state` 记每模块状态(进行/完成/需修)，中断后从未完成模块继续。
