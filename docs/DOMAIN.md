# ProductFlow 领域与业务流程

> 由 understand-anything 领域分析产出（`.understand-anything/domain-graph.json`：**6 域 · 21 流程 · 81 步骤**）。
> 架构视角见 [`ARCHITECTURE.md`](./ARCHITECTURE.md)；上手见 [`ONBOARDING.md`](./ONBOARDING.md)。

## 怎么理解「领域」

ProductFlow 是**工具**而非业务应用，所以它的「业务领域」不是订单/用户/支付，而是**「生产一个互联网产品」这件事本身的工作流**——一条核心流水线域，加五个支撑域。

```
                         ┌───────────────────────────┐
   项目生命周期  ───认领──▶│      产品生产流水线（核心）      │◀── 各阶段读写 ── 状态协议
  （建项目/预检闸门）        │  ①→②→③→④→⑤→⑥→⑦  7 阶段    │
                         └───────────────────────────┘
                             ▲          ▲          ▲
                    ②③④供图 │      ⑥保障 │      ⑦发布 │
                   视觉设计生产      质量保障        部署上线
```

## 一、产品生产流水线（核心域）

7 个阶段各是一条业务流程，步骤取自 `pf_state.py` 的 `PHASES`（权威定义），每步在操作台步骤条上可见：

| 阶段 | 步骤流 |
|------|--------|
| **①市场调研** | 明确产品与目标用户 → 竞品搜索与罗列网址 → 风格与卖点分析 → 核心矛盾分析 → 复刻要点报告 |
| **②找参考** | 确定风格方向 → 多来源找参考 → 选定参考 |
| **③首图设计** | 生成首图候选 → 定首图·视觉基调 |
| **④页面设计** | 列出所有页面 → 完成所有页面 → 各平台版本适配 → 定稿设计方向 |
| **⑤功能与数据设计** | 功能模块清单 → ER 图 → 数据层/表结构 → API 契约 → 选定开发模板 |
| **⑥开发实现** | 脚手架搭建 → 前端实现 → 后端实现 → 测试 → 接口文档 |
| **⑦部署上线** | 选择部署目标 → 执行部署 → 冒烟测试 → 交付报告 |

> 每个阶段的详细操作手册在 `productflow/references/phase-N-*.md`——领域图里流水线步骤都挂到了对应手册。

## 二、项目生命周期（支撑域）

管理项目从「诞生」到「就绪进流水线」：

- **新建与认领项目**：提交新建请求（操作台首页）→ 检查待接单队列（`~/.productflow/pending/`）→ 认领并建目录（`~/code/<slug>/`）→ `pf_state init` 注册进全局注册表。
- **生图 key 预检闸门**：检测生图 key → 强制索取 → 写入并锁权限（`~/.config/openai/env`，600）→ 放行 ③④。**这是进入首图/页面设计前的硬闸——缺 key 不静默降级。**

---

## 三、状态协议（支撑域·脊柱）

流水线的中枢神经，贯穿所有阶段：

- **状态写入与实时呈现**：agent 写状态（`pf_state.py`，排他文件锁）→ server 组装频道 payload → WebSocket 推送 → 前端渲染。
- **choice 决策握手**：抛待确认问题（`choice ask`）→ 操作台顶部呈现 → 用户点选 → agent 阻塞解析（`choice wait`，解析 stdout JSON 判超时）。
- **inbox 双向沟通**：用户在网页留言 → agent 读未读（`inbox`）→ 逐条回应（`reply`）。

> `pf_state.py` 是**唯一状态写入口**；`server.py` 只读状态并推送。这条边界是理解「双入口如何不打架」的关键。

---

## 四、视觉设计生产（支撑域，服务 ②③④）

- **参考图收集与选稿**：写关键词计划 → 抓图去重登记 → 用户选稿（`explore select-refs`）。
- **AI 批量生图**：总结风格 → `gpt-image-2`（`gen.py`，主题×风格并发）批量出图 → 登记首图 → 无限画布拖拽对比定稿。
- **框选局部重绘（inpaint）**：框选写诉求 → 生成蒙版（`_build_inpaint_mask`）→ 单块重绘（`edit.py`）→ 逐块并存新版本（原图不动、可回退）。

---

## 五、质量保障（支撑域，服务 ⑥）

两套验证体系：

- **三层自动化测试**：沙箱隔离（覆盖 `HOME`）→ CLI 状态机单测 → HTTP+WS 接口测试 → 浏览器 e2e 旅程（缺 playwright 自动 skip）。
- **对抗式 e2e 双 loop**（`productflow/tests/e2e-adversarial/`）：
  - **Loop1 发现**：轮换 8 个 persona → 跑业务旅程（`harness.py`，独立沙箱 server+浏览器）→ 记录问题（`findings.jsonl`）→ 常驻只发现、不改码。
  - **Loop2 修复**：读未修问题（`findings.py open`）→ 复现并修复 → 沙箱回归 → 标记 fixed/wontfix 并附 commit。
  - 两进程靠 `findings.py` 的 `fcntl` 跨进程锁安全并发。

---

## 六、部署上线（支撑域 = ⑦阶段能力）

多目标发布：

- **Web 多路径部署**：选 Web 路径（Cloudflare Pages / Workers+D1 / 单机 Ubuntu）→ 执行部署 → 线上冒烟 → 交付报告。
- **原生 App 与桌面上架**：iOS TestFlight → Android 内测（Google Play / 蒲公英）→ 桌面 Tauri 打包（.dmg/.msi/.AppImage）→ 冒烟（**均停在提审前**）。
- **部署凭证注入**：凭证粘贴框解析 → 写入 `~/.productflow/secrets/<id>.env`（600）→ ⑦被触发时注入环境变量（`$PF_SSH_HOST` 等，绝不进日志/产物）。

---

## 域间关系（9 条 cross_domain 边）

- **项目生命周期 → 产品生产流水线**：认领+预检通过后才进入 7 阶段。
- **状态协议 ↔ 所有阶段**：每个阶段的每一步都通过状态协议写入/呈现。
- **视觉设计生产 → ②③④**：为找参考/首图/页面阶段提供图像生产能力。
- **质量保障 → ⑥**：开发实现阶段的测试与对抗验证。
- **部署上线 = ⑦**：部署域即第七阶段的能力展开。
